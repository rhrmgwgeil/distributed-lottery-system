# 任務目標
請幫我優化 `PrizeServiceImpl.java` 與 `ActivityServiceImpl.java`。
目前系統在「新增/修改獎品」以及「修改活動」時，僅更新了 MySQL 資料庫，卻沒有同步更新 Redis 快取。這會導致營運人員在活動進行中動態調整時，前端抽獎演算法無法即時套用新規則，甚至發生沒有初始庫存的錯誤。
我們需要導入 **Cache-Aside / Write-Through** 策略，並加上嚴格的機率加總防呆機制。

# 具體修改要求

## 1. 重構 `PrizeServiceImpl.java`
請在 `createPrize` 與 `updatePrize` 方法中加入以下邏輯：
- **機率防呆校驗 (`validateProbabilitySum`)**：
  - 在寫入 DB 之前，必須撈出該 `activityId` 所有的現有獎品。
  - 將所有現有獎品的 `probability` 與即將寫入的 `newProbability` 加總（若為 update，則需先排除當前修改目標的舊機率）。
  - 若總和大於 100，拋出 `IllegalArgumentException`，拒絕操作。
- **庫存與機率清單同步 (`syncPrizeToRedis`)**：
  - 成功寫入 DB 後：
    1. 若 `prizeType` 為 1 (實體) 或 2 (虛擬)，將其 `stock` 寫入 Redis (使用 `opsForValue().set("prize:{id}:stock", stock)`) 進行快取預熱/同步。
    2. 撈出該活動「所有」最新的獎品清單，將其序列化為 JSON，並更新至 Redis (使用 `opsForValue().set("activity:{activityId}:prizes", prizesJson)`)，讓 `DrawAlgorithmService` 能即時拿到最新機率。

## 2. 重構 `ActivityServiceImpl.java`
請在 `createActivity` 與 `updateActivity` 方法中加入以下邏輯：
- **使用者抽獎上限同步 (`syncActivityMaxDrawsToRedis`)**：
  - 成功寫入 DB 後，將新的 `maxDrawsPerUser` 寫入 Redis (使用 `opsForValue().set("activity:{id}:max_draws", maxDraws)`)。
  - 這樣一來，`DrawValidationService` 在執行 `INCR` 檢查時，才能即時依據這個新的 Redis Key 來攔截。

## 3. 依賴注入調整
- 確保 `PrizeServiceImpl` 中有注入 `ObjectMapper` (處理 JSON) 以及 `StringRedisTemplate`。
- 確保 `ActivityServiceImpl` 中有注入 `StringRedisTemplate`。

# 預期結果
完成後，管理員不論是新增獎品、修改中獎機率、或是調整每人抽獎次數上限，都會在寫入 DB 的同一個 Transaction 中完成 Redis 的即時刷新，保證高併發讀取時的一致性與防呆安全。
